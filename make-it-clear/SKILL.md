# make-it-clear

## 功能

将当前对话中已完成的项目，用**一段连续的、逻辑承接紧密的叙述文字**，把技术路线和每一步的方法选择讲清楚。

输出不是表格、不是列表、不是分点陈述，而是一段流畅的文章，像讲故事一样把"为什么做 → 怎么做 → 为什么选这个方法而不是别的 → 结果暴露了什么新问题 → 下一步怎么接"的逻辑链完整串起来。

## 何时触发

- 用户完成了某个项目（分析、建模、开发、设计等），要求用连贯文字总结技术路线
- 用户说"讲清楚"、"方法逻辑"、"技术路线"、"把分析流程讲明白"
- 用户要求的不是代码回顾，而是**方法论层面的清晰阐述**

## 输出要求

### 结构

一段完整的叙述文字（可以分段，但不使用编号列表、表格、bullet points）。段落之间有明确的逻辑承接关系，上一段的结论是下一段的起点。

### 每个步骤必须覆盖的内容

对于路线中的每一步，叙述必须自然地包含：

1. **目的**：这一步要解决什么问题
2. **输入**：从上一步继承了什么
3. **方法/工具**：用了什么技术
4. **为什么选这个方法**：对比替代方案，说明当前选择的理由
5. **输出**：产出了什么，这个产出如何驱动下一步

这五点不需要逐一列出，但必须在流畅的文字中让读者自然接收到全部信息。

### 写作原则

- **逻辑链可见**：每一步的出现必须有原因——要么是上一步的结果暴露了新问题，要么是当前步骤的局限性需要下一步来弥补。禁止出现"然后我们做了 X"而没有解释"为什么需要做 X"。
- **用具体数字**：用实际数据说话（如"63677 个基因过滤到 17199 个"），而非抽象描述（如"过滤了大量基因"）。
- **方法选择有对比**：提到方法时，简要说明为什么不用替代方案。例如"用 Z-score 而非直接比较原始值，因为不同基因的量级差异达数个数量级"。
- **鞭辟入里**：不停留在"做了什么"，要揭示"为什么这样做是必要的"以及"这样做隐含了什么假设"。
- **自然流畅**：读起来像一篇好的方法论文章，而非操作手册。

### 禁止事项

- 不使用编号列表或 bullet points 作为主要组织形式
- 不使用表格
- 不逐条罗列步骤（"第一步…第二步…"）
- 不重复代码细节（如变量名、函数参数）
- 不写空泛的过渡句（如"接下来我们进入下一步"）

## 工作流程

1. 回顾当前对话，识别用户完成的项目的完整技术路线
2. 梳理每一步的目的、输入、方法、理由、输出
3. 找到步骤之间的逻辑承接关系（这一步的结果如何暴露下一步的必要性）
4. 用连贯的叙述文字一次性写出完整的技术路线阐述
5. 检查：是否每一步都回答了"为什么这样做"？逻辑链是否完整？是否有断裂的地方？

## 输出格式

```markdown
[一段或多段流畅的叙述文字，完整阐述技术路线和方法选择逻辑]
```

无标题、无列表、无表格。纯叙述。
